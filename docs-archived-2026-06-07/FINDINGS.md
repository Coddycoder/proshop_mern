# Quality review — top findings

Senior review of `proshop_mern` covering hotspots, edge cases, outdated deps, hardcoded values, and dead code. Categories per finding noted in **Cat.** Sorted by risk (highest first). All references use `file:function` form (line numbers omitted intentionally).

| # | Где | Что не так | Cat. | Уровень | Как исправить |
|---|---|---|---|---|---|
| 1 ✅ | `backend/controllers/orderController.js` → `addOrderItems` | Сервер сохраняет `itemsPrice / taxPrice / shippingPrice / totalPrice` как пришли с клиента — `orderModel` их даже не пересчитывает. Любой пользователь может оформить заказ на $0.01, и payment-результат от PayPal проверяется по `req.body.totalPrice`. Это полный bypass ценообразования. Дополнительно: проверка `if (orderItems && orderItems.length === 0)` пропускает случай `orderItems === undefined`, а после `throw` стоит мёртвый `return`. | EDGE / HOTSPOT | 🔴 critical | Загружать товары из БД по `orderItems[].product`, пересчитывать `itemsPrice / taxPrice / shippingPrice / totalPrice` на сервере (там же, где сейчас они зашиты в `PlaceOrderScreen`), игнорируя клиентские значения; добавить явную проверку `!orderItems || orderItems.length === 0`. **Исправлено в [`b7d6b09`](https://github.com/Coddycoder/proshop_mern/commit/b7d6b09a7440fe0740b61f1e55a1cb39490153db).** |
| 2 | `package.json` (root) — все backend-зависимости | `jsonwebtoken@8.5.1` имеет публично известные RCE-CVE (CVE-2022-23529 и серия 2022-23539/40/41), `multer@1.4.2` — CVE-2022-24434 (DoS), `mongoose@5.10.6` вышел из поддержки и тянет deprecated-флаги (`useNewUrlParser`, `useCreateIndex`), `bcryptjs@2.4.3`, `express@4.17.1` — всё за пределами security-окна. `npm audit` отчётом ранее показал 16 critical / 60 high. | DEPS | 🔴 critical | Поднять минимум: `jsonwebtoken@^9`, `multer@^2`, `mongoose@^8` (и параллельно — убрать deprecated-опции в `config/db.js`), `bcryptjs@^3`, `express@^4.21`/`^5`. Сначала JWT и multer как самые опасные; mongoose апгрейд потребует адаптации `Schema`/коллбэков. |
| 3 | `backend/models/userModel.js` → `userSchema.pre('save', ...)` | После `next()` в ветке `!isModified('password')` отсутствует `return` — функция продолжает выполняться и **повторно хеширует уже захешированный пароль**. На любом `user.save()` где менялся, например, только `name` или `email` (а у вас именно так в `updateUserProfile` / `updateUser`), пароль превращается в `bcrypt(bcrypt(password))`, и следующий `matchPassword` гарантированно вернёт false → пользователь теряет доступ. | EDGE / HOTSPOT | 🔴 critical | Заменить `next()` на `return next()` (и заодно завернуть основной блок в `try/catch` с `next(err)`, как принято в Mongoose-хуках). |
| 4 | `backend/middleware/authMiddleware.js` → `protect` | После `req.user = await User.findById(decoded.id).select('-password')` нет проверки на `null`. Если пользователь удалён, но JWT всё ещё валиден (срок 30 дней), `req.user` будет `null`, и любой контроллер, обращающийся к `req.user._id` (например, `getMyOrders`, `addOrderItems`, `createProductReview`), упадёт с `Cannot read property '_id' of null` вместо корректного 401. | EDGE | 🟡 medium | После `findById` проверять `if (!req.user) { res.status(401); throw new Error('Not authorized, user no longer exists') }` перед `next()`. |
| 5 | `frontend/src/screens/PlaceOrderScreen.js` (тело компонента) | Прямое мутирование redux-state во время рендера: `cart.itemsPrice = ...`, `cart.shippingPrice = ...`, `cart.taxPrice = ...`, `cart.totalPrice = ...`. Сама же логика расчёта (порог бесплатной доставки = 100, ставка налога = 0.15, page size = 10 в `productController.getProducts`, top-N = 3 в `getTopProducts`) — магические числа, разбросанные по коду, и часть из них дублируется в `OrderScreen`. Мутация `cart` под капотом полагается на reference-equality в reducer, и при будущей миграции на RTK / immer сразу сломается. | HOTSPOT / HARDCODED | 🟡 medium | Считать суммы в `useMemo` локально (или вынести в селектор/хелпер `pricing.js`), не трогая объект из `useSelector`. Параллельно вынести магические числа (`FREE_SHIPPING_THRESHOLD`, `TAX_RATE`, `PAGE_SIZE`, `TOP_PRODUCT_LIMIT`) в `backend/config/pricing.js` + frontend-зеркало или endpoint, чтобы они не разъезжались. |
| 6 | `backend/controllers/orderController.js` → `updateOrderToPaid` | Тело запроса от PayPal используется без null-checks: `req.body.payer.email_address`. Если PayPal вернёт payload без `payer` (или интегратор подменит pay-flow / отправит ручной PUT), Node упадёт `Cannot read property 'email_address' of undefined`, `express-async-handler` отдаст 500. Заодно эндпоинт никак не валидирует, что `req.body.id`/`status` действительно от PayPal — любой авторизованный пользователь может PUT'ом отметить **чужой** заказ оплаченным (кроме того, что находит его по id). | EDGE | 🟡 medium | Опционально читать поля (`req.body.payer?.email_address ?? null`); ограничить эндпоинт владельцем заказа (`order.user.equals(req.user._id)`); а в идеале верифицировать оплату серверным вызовом PayPal API по `req.body.id`, не доверяя клиентскому payload. |
| 7 | `frontend/package.json` — стек целиком | `react@16.13.1` + `react-dom@16.13.1` (EOL, нет security-патчей с 2022), `react-router-dom@5.2` (нет v6 features, перевод фреймворков на v6/v7), `redux@4.0.5` без Redux Toolkit, `axios@0.20.0` (CVE-2023-45857 — credential leakage; CVE-2024-39338 — SSRF), `react-paypal-button-v2` заброшен с 2021. Frontend-CVE менее остры, чем backend, но `axios` тут летает с JWT в `Authorization`. | DEPS | 🟡 medium | Минимум: бамп `axios@^1.7` (мелкая правка, API совместим). Полный апгрейд React/Router/Redux — это уже миграция, и апстрим сам ушёл на `proshop-v2` (Redux Toolkit + react@18+); если оставаться на этой ветке, хотя бы axios починить. |
| 8 | `frontend/src/actions/userActions.js` и `orderActions.js` (все thunks) | 7+ почти идентичных thunk'ов: `dispatch(REQUEST)` → `axios` с `Authorization: Bearer ${userInfo.token}` → `dispatch(SUCCESS)` или catch с проверкой `message === 'Not authorized, token failed'` → `dispatch(logout())`. Логика клонирована, и любая будущая правка (например, добавить refresh-token) потребует править N мест. | HOTSPOT | 🟢 cosmetic | Вынести общий `apiClient.js` (axios instance) с request-interceptor для `Authorization` и response-interceptor для 401 → `dispatch(logout())`. Thunks тогда сводятся к 5–6 строкам. |
| 9 | `backend/utils/generateToken.js` + `backend/models/userModel.js` (`pre('save')`) | Магические значения: `expiresIn: '30d'` (TTL JWT), `bcrypt.genSalt(10)` (cost factor). Прямо в коде, не выводятся в `.env`/конфиг — нельзя укоротить TTL после инцидента или поднять cost factor под современные CPU без правки исходников. | HARDCODED | 🟢 cosmetic | Прокинуть `process.env.JWT_EXPIRES_IN` (с дефолтом `30d`) и `process.env.BCRYPT_SALT_ROUNDS` (с дефолтом 10), задокументировать в README рядом с остальными env. |

> **DEAD CODE** в отдельную строку не выношу — единственный найденный (`return` после `throw` в `addOrderItems`) учтён в строке #1 как побочный пункт. Других закомментированных блоков `grep` по `backend/` и `frontend/src/` не нашёл.
